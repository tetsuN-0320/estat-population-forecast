## コーディング上の方針

### 言語・スタイル
- **Python バージョン**: 3.11 以上
- **コードフォーマッタ**: black
- **リンター**: ruff
- **型ヒント**: 関数の引数・戻り値には可能な限り付与（mypy対応）
- **docstring**: Google スタイル（日本語可）
- **コメント**: 日本語OK。なぜそうするかの「why」を残す
- **変数名・関数名**: 英語スネークケース（`fetch_population_data` 等）
- **クラス名**: 英語パスカルケース（`EstatClient` 等）

### 設計原則
- `src/` と `notebooks/` は分離する。再利用ロジックは必ず `src/` に置き、ノートブックは `from src.xxx import yyy` で import
- データ取得は SQLite キャッシュで再実行コストを抑える
- 環境変数（API キー）は `.env` で管理し、絶対にコミットしない
- フロント用の生成物（JSON）はバックエンドで集計してから書き出す

### テスト
- 主要な関数には `pytest` でユニットテストを追加
- API クライアントはモックでテスト、実 API は `pytest -m integration` のような分離マーキング推奨

## これまでの議論の経緯（要約）

1. **ポートフォリオの目的**: 業務委託受注のためのスキル証明。データアナリストとしての信頼感醸成
2. **第1作の方向性検討**: API取得→分析→可視化の流れを示すデータ分析プロジェクト。複数候補から3案に絞り込み（e-Stat 人口予測 / ふるさと納税 / Stack Overflow 学習者分析）
3. **第1案の選定理由**: APIが公的・無料・安定、予測モデル＋地理情報＋時系列のコア技能を同時アピール、実装ボリュームが中程度で初作品としてリスクが低い
4. **第2作・第3作の温存**: 第1作の枠組みを再利用することで2作目以降の実装期間を3割短縮する想定
5. **VS Code への移行**: コーディング段階に入るため Cowork から Claude Code に作業環境を切り替え
